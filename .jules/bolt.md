## 2024-07-04 - Unnecessary Meshgrid Recomputation in Batch Stitching
**Learning:** The desktop GUI application was calling `Gear360Stitcher.update_maps()` for every single image during a simple-mode batch process. Since `update_maps()` calculates computationally heavy spherical projections and blending weights using `numpy.meshgrid` and trigonometric functions, re-running this with the exact same output dimensions and camera alignment (yaw, pitch, roll) was causing severe performance degradation. This is a common performance bottleneck in media processing apps where transformation matrices are not cached across identical frames/images.
**Action:** When working on array-based remapping/stitching, ensure that computationally expensive mapping matrices (`map_x`, `map_y`) and blend masks are cached and only recomputed if the input configuration parameters (resolution, alignment offsets) change.

## 2024-07-26 - Float64 Array Promotion in Blending
**Learning:** In Python/NumPy, combining a 3-channel image (loaded via OpenCV as `uint8`) with floating-point blending weights (e.g., from `np.linspace`) implicitly promotes the entire array to `float64` during multiplication (`img1 * w1`). This causes a severe performance bottleneck during image stitching operations due to massive memory allocation and floating-point math overhead, even when the final output is immediately cast back to `uint8`.
**Action:** Always pre-scale floating-point blend weights to a `uint16` range (e.g., `(w * 256).astype(np.uint16)`) and cast the input image to `uint16` before combining. Use bitwise shifts (`>> 8`) to normalize the result before casting back to `uint8`. This limits the operation to fast fixed-point integer arithmetic and prevents `float64` promotion.
